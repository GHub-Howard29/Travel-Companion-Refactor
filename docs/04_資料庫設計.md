# 二、資料儲存架構

Travel Companion 採用 Offline First（離線優先）架構。

資料依用途儲存在不同位置。

```text
Travel Companion
        │
        ▼
     Storage Layer
        │
        ├── localStorage
        ├── IndexedDB
        └── Supabase
```

所有資料皆透過 Storage Layer 存取。

Travel Tool 不直接操作資料來源。

資料同步依功能決定。

目前規劃如下：

| 功能 | 本機 | 雲端 |
|------|------|------|
| 本機帳本 | ✓ | ✗ |
| 雲端帳本 | ✓ | ✓ |
| 共同清單 | ✗ | ✓ |
| 私人清單 | ✗ | ✓ |
| 旅行資訊 | ✗ | ✓ |

同步流程由 Foundation 統一管理。

Travel Tool 不直接處理同步。

# 七、Trip 雲端資料表

V3-1 Trip 雲端資料表 SQL：

```text
docs/sql/003_trip_cloud_schema.sql
```

執行後可用以下只讀 SQL 驗證：

```text
docs/sql/004_trip_cloud_validation.sql
```

目前定案使用：

| 資料表 | 用途 |
|------|------|
| trips | 旅程基本資料與 Trip detail content |
| admin_users | `super_admin` / `trip_editor` 權限來源 |

`trips` 目前保存：

- `id`
- `title`
- `departure_date`
- `participants`
- `currency_config`
- `sidebar_config`
- `content`
- `created_by`
- `created_at`
- `updated_at`

前端讀取旅程資料時合併三種來源：

```text
public/trips/*.json seed
Supabase trips
localStorage fallback / override
```

同步規則：

- `updatedAt` / `updated_at` 較新的版本勝出。
- App 內新增 / 編輯會先寫 localStorage，再嘗試同步 Supabase。
- Supabase 同步成功後，會把資料庫回傳的 `updated_at` 回寫到本機 cache。

Trip RLS 原則：

- Guest 可讀 `trips`。
- `super_admin` 可新增 / 編輯 / 刪除旅程。
- `trip_editor` 可編輯被指派的旅程。
- 既有 JSON seed 旅程第一次被 `trip_editor` 編輯時，允許該 `trip_editor` 建立同 id 的 `trips` row。

`admin_users` 權限原則：

- `super_admin` 可管理 `trip_editor` 名單。
- `trip_editor` 可讀自己被指派旅程的 editor 名單，但不可修改。
- 一般使用者可讀自己的權限 row。

目前進度：

- 前端已完成 App 內新增 / 編輯旅程第一版。
- SQL schema / validation 已建立於 repo。
- 尚待 Product Owner 在 Supabase SQL Editor 執行與人工驗證。

# 八、Checklist 雲端資料表

V3-1 Checklist 雲端資料表 SQL：

```text
docs/sql/001_checklist_cloud_schema.sql
```

執行後可用以下只讀 SQL 驗證：

```text
docs/sql/002_checklist_cloud_validation.sql
```

目前定案使用：

| 資料表 | 用途 |
|------|------|
| checklists | 共同 / 私人確認清單主檔 |
| checklist_items | 確認清單項目 |

`checklists.scope`：

- `shared`：共同檢查清單
- `private`：私人確認清單

私人確認清單 ownership：

本機 localStorage：

```text
userEmail + tripId
```

雲端同步：

```text
owner_user_id = auth.uid()
trip_id = 目前旅程 id
```

權限來源：

```text
admin_users
```

目前程式與 SQL 皆依 Supabase `admin_users` 判斷：

- `super_admin`
- `trip_editor`
- `trip_id`

新增旅程或新增 `trip_editor` 時，只需維護 Supabase `admin_users` 資料，不需要修改程式。

私人確認清單 RLS 原則：

- 所有私人清單讀寫皆限制 `owner_user_id = auth.uid()`。
- `super_admin` 不可讀取其他使用者的私人確認清單。
- `trip_editor` 不可讀取其他使用者的私人確認清單。
- `user` 目前只使用本機私人確認清單，不同步雲端。
- `trip_editor` / `super_admin` 可同步自己的私人確認清單到雲端。
- 前端本機 item id 以 `checklist_items.client_item_id` 保存，避免用 label 猜測雲端 row。
- `checklist_items_one_client_item_per_checklist` 確保同一清單內 `client_item_id` 不重複。

共同檢查清單 RLS 原則：

- `guest` / `user` 可讀。
- `trip_editor` 依 `trip_id` 可更新。
- `super_admin` 可更新所有 Trip 的共同檢查清單。

目前進度：

- SQL schema 已執行並完成人工驗證。
- 私人確認清單最小雲端同步已接上。
- 共同檢查清單仍使用 Trip JSON seed + localStorage 勾選進度，下一步再接 shared checklist 雲端同步。

# 九、資料關聯

```text
Trip
│
├── Members
│
├── Expenses
│   └── Attachments
│
├── Checklists
│   └── Checklist Items
│
└── References
    └── Files
```

所有 Travel Tool 的資料皆依附於 Trip。

旅程刪除時，相關資料應一併清除。

例如：

- 預約紀錄
- 行李清單
- 旅行日誌

並遵循既有資料架構。

避免修改既有資料表設計。

---

# 十一、設計原則

資料庫設計遵循以下原則：

- Trip（旅程）為資料核心。
- Storage Layer 統一管理資料存取。
- Travel Tool 不直接操作資料來源。
- 優先擴充，不修改既有架構。
- 保持低耦合、高可維護性。
