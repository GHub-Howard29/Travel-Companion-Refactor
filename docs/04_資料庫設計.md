# 一、目的

本文件說明 Travel Companion V3 的資料儲存架構。

主要描述：

- 資料儲存位置
- 資料同步方式
- 資料表設計

本文件不討論產品功能，也不討論程式實作。

---

# 二、資料儲存架構

Travel Companion 採用 Offline First（離線優先）架構。

資料依用途儲存在不同位置。

```text
Travel Companion
        │
        ▼
     Storage Layer
        │
        ├── localStorage
        ├── IndexedDB
        └── Supabase
```

所有資料皆透過 Storage Layer 存取。

Travel Tool 不直接操作資料來源。

# 三、本機資料

目前使用：

## localStorage

用途：

- 使用者設定
- UI 設定
- 快取資訊

特色：

- 容量小
- 存取快速
- 不同步

---

## IndexedDB

用途：

- 本機帳本
- 附件
- 離線資料
- 暫存同步資料

特色：

- 容量大
- 支援附件
- 支援離線
- 可與雲端同步

# 四、雲端資料

目前使用：

Supabase

用途：

- 旅程
- 成員
- 雲端帳本
- 共同清單
- 私人清單
- 旅行資訊
- 附件

特色：

- 多人共同使用
- 自動同步
- 權限管理

# 五、資料同步

資料同步依功能決定。

目前規劃如下：

| 功能 | 本機 | 雲端 |
|------|------|------|
| 本機帳本 | ✓ | ✗ |
| 雲端帳本 | ✓ | ✓ |
| 共同清單 | ✗ | ✓ |
| 私人清單 | ✗ | ✓ |
| 旅行資訊 | ✗ | ✓ |

同步流程由 Foundation 統一管理。

Travel Tool 不直接處理同步。

目前規劃如下：

| 資料表 | 用途 |
|------|------|
| trips | 旅程資料 |
| trip_members | 旅程成員 |

# 七、Checklist 雲端資料表

V3-1 Checklist 雲端資料表 SQL：

```text
docs/sql/001_checklist_cloud_schema.sql
```

執行後可用以下只讀 SQL 驗證：

```text
docs/sql/002_checklist_cloud_validation.sql
```

目前定案使用：

| 資料表 | 用途 |
|------|------|
| checklists | 共同 / 私人確認清單主檔 |
| checklist_items | 確認清單項目 |

`checklists.scope`：

- `shared`：共同檢查清單
- `private`：私人確認清單

私人確認清單 ownership：

```text
owner_user_id = auth.uid()
trip_id = 目前旅程 id
```

也就是：

```text
userId + tripId
```

權限來源：

```text
admin_users
```

目前程式與 SQL 皆依 Supabase `admin_users` 判斷：

- `super_admin`
- `trip_editor`
- `trip_id`

新增旅程或新增 `trip_editor` 時，只需維護 Supabase `admin_users` 資料，不需要修改程式。

私人確認清單 RLS 原則：

- 所有私人清單讀寫皆限制 `owner_user_id = auth.uid()`。
- `super_admin` 不可讀取其他使用者的私人確認清單。
- `trip_editor` 不可讀取其他使用者的私人確認清單。
- `user` 目前只使用本機私人確認清單，不同步雲端。
- `trip_editor` / `super_admin` 可同步自己的私人確認清單到雲端。

共同檢查清單 RLS 原則：

- `guest` / `user` 可讀。
- `trip_editor` 依 `trip_id` 可更新。
- `super_admin` 可更新所有 Trip 的共同檢查清單。

# 八、資料關聯

```text
Trip
│
├── Members
│
├── Expenses
│   └── Attachments
│
├── Checklists
│   └── Checklist Items
│
└── References
    └── Files
```

所有 Travel Tool 的資料皆依附於 Trip。

旅程刪除時，相關資料應一併清除。

# 九、未來規劃

未來新增功能時：

例如：

- 預約紀錄
- 行李清單
- 旅行日誌
- 保險資訊

若需要雲端儲存，

則新增對應資料表。

並遵循既有資料架構。

避免修改既有資料表設計。

---

# 十、設計原則

資料庫設計遵循以下原則：

- Trip（旅程）為資料核心。
- Storage Layer 統一管理資料存取。
- Travel Tool 不直接操作資料來源。
- 優先擴充，不修改既有架構。
- 保持低耦合、高可維護性。
