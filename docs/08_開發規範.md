# Travel Companion 開發規範

> Version：V3.0
>
> 最後更新：2026-07

---

# 一、目的

本文件定義 Travel Companion 專案的開發流程、設計原則與合作方式。

所有功能開發、重構與維護皆以本文件為最高準則。

---

# 二、開發理念

## Product First（產品優先）

先思考產品，再思考程式。

所有設計皆應以使用者需求與產品架構為出發點，而非 React、TypeScript 或其他技術。

---

## Trip First（Trip 為核心）

Trip 為整個產品的核心。

所有功能皆屬於 Trip，不允許獨立於 Trip 存在。

---

## Foundation First（共同能力優先）

共同能力集中管理。

各功能僅負責自己的商業邏輯。

---

## Simple First（簡單優先）

優先選擇：

- 可讀
- 可維護
- 易於擴充
- 不過度設計

避免為了未來可能不會出現的需求而增加複雜度。

---

# 三、開發流程

所有功能皆遵循以下流程：

1. 分析需求
2. 架構設計
3. 確認方案
4. Coding
5. Build
6. Test
7. Commit
8. 視 Product Owner 要求集中整理文件

未完成以上流程，不視為功能完成。

---

# 四、討論原則

一次只討論一個主題。

一次只完成一個 Step。

避免同時修改多個模組。

---

# 五、角色分工

## Product Owner（使用者）

負責：

- 功能需求
- 使用流程
- 使用情境
- 產品方向

不需要負責程式架構。

---

## Software Architect（ChatGPT）

負責：

- 系統架構
- 模組規劃
- 資料流
- 程式設計
- 重構策略

所有技術討論皆應轉換成產品角度說明。

---

# 六、程式設計原則

遵循 Single Responsibility Principle。

每個 Module 僅負責一項職責。

每個 Function 僅完成一件事情。

避免大型檔案。

避免重複程式。

避免 Tool 彼此直接依賴。

---

# 七、新增功能原則

新增功能時：

優先新增 Module。

避免修改既有 Module。

若必須修改既有程式，應確認是否為架構設計問題。

---

# 八、Build 規範

每完成一個 Step：

必須執行：

npm run build

Build 成功後才能進入下一步。

---

# 九、Commit 規範

每個 Step 完成：

建立一次 Commit。

Commit 使用繁體中文。

Commit 應描述本次完成內容。

---

# 十、文件規範

文件更新採集中整理原則。

平常實作功能、修 Bug、重構時：

- 先完成程式碼、Build 與必要測試。
- 不因每個小 Step 主動更新文件。
- 不把文件同步作為每次小修改的固定工作。

只有在 Product Owner 明確要求時，才集中整理：

- 整理文件
- 整理交接文件
- 更新 README
- 準備 Commit
- 要求 Commit
- 其他直接指定文件維護的指令

集中整理時，依實際影響更新：

- 架構藍圖
- 模組架構說明
- 產品開發路線圖
- 開發進度
- 版本更新紀錄

若程式修改會讓既有文件中的定案規則明顯誤導後續開發，應先提醒 Product Owner，再由 Product Owner 決定是否立即更新文件。

---

# 十一、程式註解規範

採用區塊式中文註解。

例如：

// 負責權限判斷//

...

// Trip 狀態管理//

避免逐行註解。

程式命名維持英文。

註解使用繁體中文。

---

# 十二、重構原則

重構不得改變功能。

每次重構：

- Build
- 測試
- Commit

確認正常後再進入下一步。

---

# 十三、長期目標

打造一套：

- 易閱讀
- 易維護
- 易測試
- 易擴充
- 五年後仍容易理解

的 Travel Companion V3 系統。
